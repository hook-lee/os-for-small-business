# 워크스페이스 v1 — 디자인 문서

- **작성일**: 2026-05-20
- **최종 업데이트**: 2026-05-20 (라파 필라테스 실데이터 분석 반영)
- **상태**: 디자인 확정, 구현 계획 단계로 진행 예정
- **유형**: 개인 프로젝트 (선물용 도구)

---

## 1. 요약

여자친구 "유진"이 운영하는 **라파 필라테스**(2024-02 창업)의 운영을 돕는 단일 사용자용 웹 대시보드.
이미 운영자가 2년 이상 굴려온 **구글 시트 + Looker Studio** 시스템 위에, **세금 시뮬레이터(부가세·종소세, 청년창업감면 반영)** 와 **권장 예비비 안내**를 얹는다.
새로운 입력 도구를 만들지 않는다 — 익숙한 시트는 그대로 두고, 그 위에 "미래 정보"를 더하는 레이어로 동작한다.
**운영자의 실데이터(2024-02-22 ~ 2026-01-17, 2,566 거래)** 가 시뮬레이터 검증 데이터셋으로 확보됨.

---

## 2. 배경 & 동기

운영자(유진)는 이미 다음을 잘 운용 중이다 (실데이터 확인):

- 일별 거래 ledger: 날짜·계정과목·금액·수단·상대방·사람(클라이언트/직원). 2024-02-22부터 거의 매일 입력.
- 시트 29개 (스터디 종합 + 전체 통합 + 월별 27개 + 1개 카피).
- "전체 통합" 시트에 2,566개 거래 누적.
- "스터디 종합" 시트에 6가지 절세 가이드 (사용자가 직접 정리한 라파 전용 노하우).
- 월별 정산: 매출/지출/영업이익/순수익 자동 집계.
- Looker Studio 연도별 KPI + 월별 추세 차트 + 사용자가 단 주석("*6월 여름 이벤트", "*3월 오픈 전!" 등).
- 사업비/생활비 영역 수동 분리 (영업이익 라인 기준).
- 종소세·부가세 대비 월 180만원 예비비 자동 적립.

PDF 대시보드의 검증된 KPI:
- 2024년: 매출 1.02억, 지출 -8,396만, 순이익 +1,808만, 유진 급여 3,600만
- 2025년: 매출 1.18억, 지출 -1.01억, 순이익 +1,678만, 유진 급여 3,100만 (현금 제외 매출 1.02억 → 현금 매출 약 1,600만)
- 2026년 (1월): 매출 641만, 지출 -321만, 순 +320만

운영자가 빠뜨리고 있는 4가지 — v1이 이 빈칸을 채운다:

- **세금 예상치를 미리 못 본다** — 분기 부가세 신고 직전에서야 확인.
- **권장 예비비가 정적** — 월 180만 고정이지만, 매출 변동·청년창업감면 적용 여부에 따라 동적이어야 함.
- **납부일을 챙길 외부 시스템 없음** — 캘린더 따로 운영.
- **절세 기회를 놓침** — 특히 만 28세 + 2024-02 창업 + 필라테스업 = **청년창업중소기업 특별세액감면** 대상일 가능성 매우 큼 (5년간 50~100% 종소세 감면). 이 한 항목으로만 종소세가 2배 차이 날 수 있음.

## 4. v1 스코프

### In

1. **구글 시트 자동 읽기** — 운영자 본인의 시트를 Google Sheets API로 읽기 전용 접근
2. **카테고리 정규화 레이어** — 시트의 47개 변형(오타·동의어·공백)을 30개 표준 카테고리로 자동 매핑
3. **부가세 시뮬레이터** — 분기별 매출세액 − 매입세액 (owner draw·예비비 제외)
4. **종소세 시뮬레이터** — 연간 누적 영업이익 → 누진세율 + 기본 공제 + **청년창업감면 토글 (0/50/100%)**
5. **권장 예비비** — 현재 매출 페이스 + 청년창업감면 반영하여 매월 동적 추천
6. **납부일 D-day 알림** — 3·6·9·12월 부가세, 5월 종소세
7. **대시보드** — 기존 Looker 레이아웃을 계승 + 세금 카드 + **전년 동월 대비 비교**
8. **월별 주석 메모** — 사용자가 PDF에 단 "*6월 여름 이벤트" 같은 주석을 디지털로 보존
9. **절세 액션 카드** — 라파 시트 "스터디 종합"의 6개 절세 가이드를 카드화 (청년창업감면, 사업용 카드, 공과금 사업자 전환, 경조사 증빙, 차량비, 소모품)
10. **실데이터 fixture & 백테스트** — 2566건 거래 + PDF KPI 정답지로 시뮬레이터 정확도 검증

### Out (v2 이후)

- 거래 입력 UI (계속 구글 시트로)
- 자동 데이터 연동(코드에프·홈택스 직결)
- 모바일 앱
- 알림(이메일·푸시·카카오톡)
- 멀티 유저 / 인증 / 결제
- 회원 관리·전환율·재등록율 분석
- 프리랜서 시간/급여
- AI 기반 이벤트 추천 + 시즌성 자동 감지 (룰 기반 v1, AI 기반 v2)
- 카테고리 자동 학습 (사용자가 토글한 결과 누적해서 추천 개선)

## 5. 아키텍처

```
[데이터]                [백엔드]               [프론트엔드]
 구글 시트  ─────→  Next.js API Routes  ───→  Next.js + Recharts
 (원본,             + Google Sheets             (Vercel 무료 배포)
  source of         API + 정규화 레이어
  truth)            + 세금 계산 순수 함수
                    + 5~10분 캐싱
                    ↑
                    └─ fallback: 실데이터 fixture
                       (시트 접근 실패 시 또는 데모 시)
```

- **DB 없음**: 시트가 원본. 우리는 읽기·정규화·계산·표시만.
- **인증**: 운영자 본인의 Google 서비스 계정 (시트 read-only). 앱 자체는 베이직 인증(비밀번호 1개).
- **배포**: Vercel 무료 티어 (트래픽 1명).
- **세금 계산 로직**: 순수 TypeScript 함수, 단위 테스트 + 실데이터 백테스트로 검증.
- **캐싱**: 시트 데이터 5~10분.
- **Fallback**: 시트 접근 실패 시 `tests/fixtures/real-transactions.ts`의 2,566건 실데이터로 dashboards 렌더 (앱 안 깨짐).

## 6. 데이터 모델 (실데이터 기반 확장)

### 6.1 Transaction (정규화된 거래)

```
Transaction
├ date             ISO yyyy-mm-dd
├ rawCategory      시트 원본 카테고리 (예: "임차료", " 유진 급여")
├ category         정규화 표준 카테고리 (예: "임대료", "유진 급여")
├ amount           number (부호로 매출/지출)
├ method           '카드' | '계좌이체' | '현금'
├ counterparty     비고1 (가맹점/거래처)
├ person           비고2 (사람 — 클라이언트 또는 직원)
├ classification   'business' | 'living' | 'capital' | 'owner_draw' | 'reserve'
└ memo             (선택)
```

**분류(classification) 정책**:
- `owner_draw`: "유진 급여" → owner draw로 별도 KPI. 종소세 계산에서 사업소득 차감 X.
- `reserve`: "예비비" → 별도 KPI. 실제 비용 아님, 세금용 적립.
- `capital`: "자산", "보통예금", "권리금", "사무용품" (창업 초기) → 자산성 지출, 감가상각 대상. v1에서는 KPI 표시만 하고 매년 비용 처리에서 제외.
- `business`: 사업비 (임대료·마케팅비·정기결제·세금·보험료·공과금·관리비·수수료·교육비 등).
- `living`: 생활비 (식비·품위유지비·교통비·의류비·의료비·소모품·소품·도서인쇄비·경조사비).

### 6.2 표준 카테고리 (30개) + 정규화 매핑

운영자 시트에서 발견된 47개 카테고리를 30개 표준으로 정규화:

```
임차료 → 임대료
소모품비 → 소모품
마케팅 → 마케팅비
월급 → 급여

교육 → 교육비
인쇄비 → 도서인쇄비
프린트 → 도서인쇄비
보혐료 → 보험료 (오타)
경좃비 → 경조사비 (오타)
경조선물비 → 경조사비

보험료 · 품위유지비 · 교통비 · 의류비 · 의료비 · 소품 · 도서인쇄비 ·

```

### 6.3 매출 분리

매출은 두 가지로 KPI 표시:
- `revenueTotal` — 전체 매출 (카드 + 계좌이체 + 현금)
- `revenueCardOnly` — 카드/계좌이체 매출 (홈택스 자동 캡처 대상)
- 차이 = 현금 매출 (운영자가 별도 메모 가능)

## 7. 세금 시뮬레이션 로직

### 7.1 부가세 (분기별)

```
입력: 해당 분기의 모든 거래
─────────────────────────────────────
매출세액 = Σ(매출 × 10/110)
매입세액 = Σ(공제대상 거래 × 10/110)
공제대상 판정:
  classification == 'business' AND
  수단 IN ('카드','계좌이체') AND
  계정과목 NOT IN ('접대비','경조사비')
  AND owner_draw·reserve·capital 제외

예상부가세 = 매출세액 − 매입세액

라파 실데이터 적용 시 (필라테스 = 인적용역):
  매입 거의 없으므로 매출세액 ≈ 부가세
  2025년 추정 부가세 ≈ 매출 1.18억 × 10/110 ≈ 1,073만/년
```

### 7.2 종소세 (연간, 청년창업감면 포함)

```
입력: 연초~asOfDate 거래 + 잔여기간 페이스 환산
─────────────────────────────────────
연환산 사업소득금액 = ((매출 − 사업비용) / 경과월) × 12
                       (owner_draw, capital, reserve 제외)
종합소득금액 = 연환산 사업소득 + 사용자 입력 기타소득
과세표준 = 종합소득 − 인적공제(150만/명) − 노란우산 − 연금저축
산출세액 = 과세표준 × 누진세율 (6/15/24/35/38/40/42/45%)
세액공제 = 표준 7만 + 사용자 입력

🆕 청년창업감면 (옵션):
  - 0% (적용 안 함, 디폴트)
  - 50% (수도권 과밀억제권역)
  - 100% (수도권 외 / 비수도권)
  
예상세액 = (산출세액 − 세액공제) × (1 − youngStartupReduction)

라파 적용 시 (예시):
  2025년 연환산 사업소득금액 ≈ 1.68천만 (PDF 순이익)
  과세표준 ≈ 1,500만
  산출세액 ≈ 110만 (감면 전)
  청년창업감면 100% 적용 시 → 0원
  50% 적용 시 → 55만
```

### 7.3 권장 월 예비비

```
권장 = (연 예상 부가세 + 연 예상 종소세 (감면 반영)) / 12

라파 실데이터 적용 시 (감면 100% 가정):
  부가세 ≈ 1,073만 + 종소세 ≈ 0 = 1,073만/년
  권장 월 예비비 = 약 89만/월

감면 0% 가정 시:
  부가세 1,073만 + 종소세 110만 = 1,183만/년
  권장 월 예비비 = 약 98만/월

⚠️ 운영자가 직관적으로 적립해온 180만/월보다 낮음.
이유: 라파의 영업이익률이 종소세 누진 구간 진입에 미달함.
운영자가 보수적으로 예비비 두고 있던 것으로 보임 — 좋은 사인.
```

### 7.4 절세 액션 카드 (스터디 종합 시트 6개 가이드 디지털화)

| 시트의 가이드 | 우리 카드 형태 | 트리거 조건 |
|---|---|---|
| 1. 사업용 신용카드 등록 (필수) | "현금 결제 비율 N%, 사업용 카드로 몰아 쓰시면 부가세 ○○만원 추가 공제 가능" | 항상 표시 + 현금 매입 발견 시 강조 |
| 2. 공과금 사업자용 전환 | "공과금 세금계산서 받고 있나요? 부가세 10% 환급 가능" | 항상 표시 |
| 3. 경조사비 증빙 관리 | "경조사비 결제 발견 — 청첩장/부고장 챙기기 (20만원 한도)" | 경조사비 카테고리 결제 발생 시 |
